## 1. プロダクト概要

### 1.1 目的
民間学童保育所の業務運営(会員管理・請求・労務・行事・会議・規程管理)を一元化するマルチテナントSaaS。保護者会主導の年次運営を、人が変わっても回るように仕組み化する。

### 1.2 ターゲット
- **顧客**: 民間学童保育所(民設民営、保護者会主導運営)
- **規模**: 1施設あたり児童最大100名規模まで対応
- **初期顧客**: ひまわりクラブ(本山南学童保育所)
- **拡大**: 半年後(2026年11月頃)を目処に他施設提供開始

### 1.3 利用者(マルチロール)
1. SaaS運営者
2. 運営委員(地域代表・学校・PTA等)
3. 施設管理者(所長)
4. 主任指導員 / 副主任指導員 / 正規指導員 / パート指導員
5. 保護者会役員(会長・副会長・書記・会計(一般)・会計(給与)・会計(労務)・会計監査・オブザーバー)
6. 行事実行委員(都度任命、時限ロール)
7. 一般保護者

---

## 2. アーキテクチャ概要

- **クラウド**: AWS サーバーレス
- **認証**: Cognito User Pool (共通プール + `org_id` 属性でテナント切り分け)
- **API**: API Gateway + Lambda
- **データストア**: DynamoDB(中心)、S3(添付・PDF・文書)
- **配信**: SES(メール通知一本)、EventBridge(締め日バッチ、ロール失効バッチ)
- **AI**: Bedrock(Claude) + 独立Lambda + S3/ベクトルDBのRAG基盤
- **IaC**: AWS CDK (TypeScript)
- **Lambda言語**: Python(指定なき場合) / TypeScript

## 3. テナント・認証・ロール

### 3.1 テナント設計
- マルチテナント(`org_id` で完全分離)
- 1テナント = 1学童保育所
- DynamoDB は全テーブルでパーティションキーに `org_id` を含めて分離

### 3.2 認証方式
- Cognito User Pool 共通プール
- カスタム属性: `org_id`(所属施設), `user_type`(staff/parent/operator)
- 1ユーザーが複数施設に所属するケースはMVP対象外(将来検討)

### 3.3 ロール設計

#### 静的ロール(階層)
| ロール | スコープ |
|---|---|
| SaaS_OPERATOR | 全テナント |
| ORG_ADMIN(施設管理者) | 当該施設の全機能 |
| RUNNING_COMMITTEE_MEMBER(運営委員) | 当該施設の閲覧+総会議決 |
| STAFF_HEAD(主任指導員) | 指導員集団のリーダー |
| STAFF_SUB_HEAD(副主任指導員) | 主任補佐 |
| STAFF_FULL(正規指導員) | 保育・出欠・日誌・連絡 |
| STAFF_PART(パート指導員) | 保育・出欠・日誌 |
| PARENT(一般保護者) | 自世帯の閲覧・申込 |

#### 動的・時限ロール(保護者会・行事)
| ロール | 任期 | 主な権限 |
|---|---|---|
| PARENT_CHAIR(会長) | 1年 | 保護者会代表、対外契約 |
| PARENT_VICE_CHAIR(副会長) | 1年 | 会長補佐、地域連携 |
| PARENT_SECRETARY(書記) | 1年 | 議事録管理 |
| PARENT_ACCOUNTING_GENERAL(一般会計) | 1年 | 一般会計、入金消込 |
| PARENT_ACCOUNTING_PAYROLL(給与会計) | 1年 | 指導員給与計算・支払 |
| PARENT_ACCOUNTING_LABOR(労務会計) | 1年 | 労災・雇用保険・厚生年金 |
| PARENT_ACCOUNTING_AUDITOR(会計監査) | 1年 | 会計監査・財務閲覧 |
| EVENT_COMMITTEE_LEAD(行事実行委員長) | 行事期間 | 当該行事の参加管理・実費精算 |
| EVENT_COMMITTEE_MEMBER(行事実行委員) | 行事期間 | 当該行事の参加者連絡・名簿閲覧 |
| MEETING_DELEGATE(会議担当) | 当該会議期間 | 市連全体会・100人集会等の参加・例会司会 |
| SIGNATURE_CAMPAIGN(署名活動担当) | 当該期間 | 署名活動の参加 |

**重要**: すべての動的ロールは `validFrom` / `validTo` を持ち、期間外は自動失効。EventBridge で日次バッチを回して期限切れを反映。

### 3.4 監査ログ
- 全エンティティに `createdBy`, `createdAt`, `updatedBy`, `updatedAt` を記録
- 編集履歴は別テーブル `AuditLog` に時系列追記(誰がいつ何を変えたか、変更前後の値)
- 監査ログは閲覧専用、削除不可

### 3.5 権限ポリシー
- **閲覧は広く、編集はロール限定**
- 例: 議事録は全保護者閲覧可、編集は書記のみ
- 例: 行事の参加者名簿は実行委員閲覧可、編集は実行委員長のみ
- ロール変更時に過去のアクセス権は遡及失効しない(過去の変更ログは保持)

## 4. 会員・世帯管理

### 4.1 世帯モデル
- `Household`(世帯): 住所・電話番号・口座振替情報
- `Member`(世帯メンバー): 世帯に紐づく個人

### 4.2 メンバーステータス
| 区分 | ステータス | 説明 |
|---|---|---|
| 児童 | `PROSPECTIVE` | 未就学だが将来入所予定 |
| 児童 | `ACTIVE` | 在籍中 |
| 児童 | `GRADUATED` | 卒所(6年生まで通った) |
| 児童 | `WITHDRAWN` | 途中退所 |
| 兄弟 | `PRESCHOOL_GUEST` | 学童には入らないがイベント参加可 |
| 兄弟 | `ALUMNI_GUEST` | 卒所後もイベント参加(中学生以上) |
| 保護者 | `PRIMARY_GUARDIAN` | 主担当(口座振替契約者) |
| 保護者 | `SECONDARY_GUARDIAN` | 副担当 |
| 連絡先 | `EMERGENCY_CONTACT` | 緊急連絡先(祖父母など) |

### 4.3 メンバー属性
- 氏名(姓・名・カナ)、続柄、生年月日、性別
- 連絡先(メール・電話・LINE等)
- アレルギー・配慮事項
- 写真公開可否(SNS・パンフレット・卒所アルバム個別)
- 緊急時対応フラグ(医療情報・かかりつけ医)
- 在籍履歴(入所年月、退所/卒所年月、学年遷移)

### 4.4 個人情報保護
- **保存期間**: 卒所後5年保持 → 自動アーカイブ(S3 Glacier等)
- **削除**: テナント管理者の明示操作のみ
- アーカイブ後は通常検索からは除外、過去歴照会機能のみで閲覧可能
- 卒所児童の写真・連絡先は別途オプトイン管理

## 5. 料金・請求・入金管理

### 5.1 料金品目(カタログ)
品目はテナントごとに自由に設定。ひまわり実態を初期テンプレとして提供:

| カテゴリ | 例 | 単価 |
|---|---|---|
| 入所金 | 入所時1回 | 20,000円(兄弟2人目以降10,000円割引) |
| 基本保育料(月額) | 低学年/4年/5年/6年 | 14,000 / 5,000 / 4,000 / 3,000 円 |
| 特別保育料 | 夏休み / 冬休み | 各 10,000円(月額1,700円相当を該当月に上乗せ) |
| 延長保育料 | 一時利用 | 200円/30分 |
| 兄弟割引 | 低学年同士 | -1,000円/月(2人目以降) |
| 単身割引 | 低学年のみ | -2,000円/月 |
| 市連会費 | 年額 | 2,400円/世帯(4月一括) |
| 保護者会費 | 年額 | 2,400円/世帯(5月一括) |
| 昼食代 | 長期休暇・臨時休校 | 350円/日 |
| 外出時品目 | ジュース/お茶等 | 150円 / 110円 等 |
| イベント料金 | キャンプ年齢区分 | 大人5000 / 学童3000 / 未就学1500 / 中学生=大人 |

### 5.2 請求書(MONTH / EVENT / VACATION / OUTING)
- 締めは月次が基本だが、EVENT / VACATION / OUTING は単位ごとに発行
- 台帳(`Ledger`)方式: 課金/返金は1行ずつ追記
- 単価変更で過去ログが壊れないよう、台帳に `unitPrice` と `amount` を記録
- 訂正は削除ではなく **訂正行(差額)** 追加で整合を取る
- 請求書スナップショットは生成時点の台帳の集約

### 5.3 配信
- 保護者ログイン画面で **PDF閲覧可**
- メール送信(SES)でも配信
- **印刷PDF**は出力しない(MVPスコープ外)

### 5.4 決済・入金管理
- **決済機能なし**(SaaSは請求書発行まで)
- 既存の口座振替(播州信用金庫東灘支店等)を維持
- 入金確認は **通帳CSV取込 or 手動入力で消込フラグ更新**
- 未収一覧レポートで滞納を把握

### 5.5 イベント実費精算
- イベント参加申込 → 見込み負担額(plannedCharge)算出
- 終了後、実費(receiptAmount)を入力 → 余剰金は参加者の `plannedCharge` 比例で返金按分、不足は同様に按分追加徴収
- **端数処理**: 合計が一致するよう最大負担者から ±1円で調整(明文化された決定的ルール)

## 7. AI支援機能(段階的拡張)

### 7.1 MVP優先(◎ 強相性)
1. **引継ぎ文書の自動生成** — 過去の同行事(ファミリーキャンプ等)の実行委員記録・収支・反省点をRAGして引継ぎ書生成
2. **議事録の要約・議題タグ・決定事項抽出** — 録音/テキストから要約
3. **保育日誌のサマリと気になる児童の検知** — 週次サマリ生成

### 7.2 第2フェーズ(○ 相性が良い)
4. 保護者問い合わせ自動応答(規程RAG)
5. シフト自動作成提案
6. 規程改定ドラフト支援
7. アレルギー・配慮事項の見落とし防止
8. 年次総括・保育総括のドラフト

### 7.3 優先度低(△)
- 滞納督促文面生成、写真整理、入所説明会Q&A

### 7.4 実装方針
- 独立Lambda + Bedrock(Claude) で疎結合
- RAG基盤(S3 + ベクトルDB)を1つ立てて文書系AIは全部その上に乗せる
- AI利用コストはテナント課金に反映(従量制を検討)

## 8. 非機能要件

### 8.1 パフォーマンス
- 1施設100名規模で快適に動作
- DynamoDBはホットパーティション回避(施設×請求単位×期間で分散)
- 読み取りパターンに合わせたGSI設計

### 8.2 セキュリティ
- 個人情報暗号化(at rest, in transit)
- ロール変更・閲覧含む全操作の監査ログ
- 個人情報保護法準拠(保存期間、削除権)
- セッション管理(Cognito)、CSRF/XSS/SQLi対策

### 8.3 可用性
- AWSマネージドサービス前提
- 単一リージョン(ap-northeast-1)、必要に応じてマルチAZ
- バックアップ: DynamoDB Point-in-Time Recovery 有効化

### 8.4 コスト
- DynamoDBオンデマンド or 適切なキャパシティ
- Lambda従量
- 想定: 1施設あたり月額数千円〜(施設数依存)

## 9. 開発・運用

### 9.1 リポジトリ・環境
- 本リポジトリ `gakudo-saas` で構築
- ひまわりクラブHP(`himawari`)とは完全分離・疎結合
- AWSアカウントも分離(本リポジトリ専用 or サブアカウント)

### 9.2 開発スタイル
- IaC: AWS CDK (TypeScript)
- Lambda: Python (一部 TypeScript)
- 開発環境: VSCode + GitHub + Claude

### 9.3 段階的実装計画
1. **Phase 0**(本書): 仕様確定
2. **Phase 1**: CDKスタック雛形、DynamoDBテーブル設計、認証基盤
3. **Phase 2**: 会員・世帯管理、料金カタログ、請求書発行(MONTH)
4. **Phase 3**: イベント・休暇・外出時の請求書、実費按分
5. **Phase 4**: 保護者画面、メール配信、通帳CSV取込
6. **Phase 5**: 指導員・労務・給与計算、シフト管理
7. **Phase 6**: 議事録・規程・お知らせ・出席・レポート
8. **Phase 7**: 総会議決Web化、AI支援機能(引継ぎ・議事録要約)
9. **Phase 8**: ひまわりクラブで実運用開始
10. **Phase 9**(6ヶ月後): 他施設提供開始、運営者向け管理画面強化
