# 学童SaaS仕様 — 対話記録

**日付**: 2026-05-11
**形式**: 対話記録 (`/talk` スキル)
**対象プロジェクト**: gakudo-saas (学童保育向け請求・会員管理SaaS)
**参考資料**: 本山南学童保育所ひまわりクラブ 第37回総会議案書 (2026年5月10日)

---

## 背景・動機

ひまわりクラブ(本山南学童保育所)の運営実態を踏まえた、**学童保育向けマルチテナントSaaS** を構築する。最初の利用施設はひまわりクラブ自身。半年程度の実運用を経て、他の民間学童保育所への販売・提供を開始する。

民間学童は保護者会主導の民設民営運営が一般的で、保護者が役員・行事実行委員になって会計・労務・行事運営を担う特殊な組織形態。指導員(正規・パートタイム)の労働契約管理、児童の保育、保護者会の民主的運営、地域・運営委員会との連携、外部監査・規程改定など、業務範囲が非常に広い。

この複雑な運営を、年次で役員が交代する保護者組織でも回せるようにする — それが本SaaSの存在意義。

---

## 問題意識

- 保護者役員が **1年で交代** するため、毎年の引継ぎコストが大きい。行事実行委員も毎年変わる。
- 指導員の **新規採用・退職が頻繁**(2025年度はパート3名入れ替わり)。労務管理・賃金計算・有給管理が手作業
- **保育料体系が複雑**(月額、夏冬特別、延長、兄弟割引、単身割引、市連会費、保護者会費、長期休暇昼食代、行事実費)
- イベントごとの実費精算・按分・余剰金返金がすべて手作業
- 議事録・規程・契約書などの文書が紙とExcelで散在。改定履歴管理が属人化
- 指導員間のコミュニケーション不足、子どもの変化への気付き遅れが課題として総括に明記されている

---

## 前提・現状認識

### 組織構造(ひまわり実態)
- **3層構造**: 運営委員会(地域代表+学校+PTA+民児協) / 保護者会(役員会+一般保護者) / 指導員(主任・副主任・正規・パート)
- 保護者会役員: 会長1, 副会長2, 書記1, 会計(一般/給与/労務)3名, 会計監査1, オブザーバー
- 任期1年・再任可。**保護者は児童在籍中に1回(2人以上で2回)以上、役員就任義務**
- 行事ごとに **実行委員(5〜9名の保護者)** が組成され、企画・集金・実費精算を担当

### 既存運用
- 保育料は **播州信用金庫東灘支店の口座振替**(毎月5日)
- 通知は紙配布・LINE・口頭の混在
- 議事録・規程・契約書は紙・Word
- 指導員給与は給与会計役員が計算、賃金規程・就業規則は毎年総会で改定

### 法令・基準
- 児童福祉法第6条の3第2項(放課後児童健全育成事業)
- 神戸市放課後児童健全育成事業の設備及び運営に関する基準を定める条例
- 神戸の放課後児童クラブ(学童保育)の基準(ガイドライン)
- 個人情報保護法

## 確定した仕様(対話の合意)

### ビジネス・テナント
- **マルチテナントSaaSとして設計**。フェーズ1: ひまわりクラブ1施設で実運用、フェーズ2(6ヶ月後): 他施設提供開始
- 想定顧客: **民間学童**、1施設あたり**最大児童100名規模**まで対応
- 認証: **Cognito共通プール + テナント属性(org_id)** でマルチテナント切り分け
- 保護者ログインあり(全機能対象、第1版から)

### ロール設計
- 階層ロール: SaaS運営者 / 運営委員 / 施設管理者(所長) / 主任指導員 / 副主任指導員 / 正規指導員 / パート指導員 / 保護者役員(会長・副会長・書記・会計各種・会計監査・オブザーバー) / 行事実行委員(都度任命) / 一般保護者
- **時限的ロール割当**: ロール有効期間(validFrom, validTo)を必ず持つ。任期満了で自動失効
- **閲覧は広く、編集はロール限定**。全データに **監査ログ(誰がいつ何を編集したか)** を持つ
- 過去のロール履歴も保持(誰が何年に役員だったか引ける)

### メンバー(世帯)設計
世帯(household)に紐づく全員を単一テーブルで保持、ステータスで切り分け:
- 児童: PROSPECTIVE(将来入所予定) / ACTIVE(在籍中) / GRADUATED(卒所) / WITHDRAWN(途中退所)
- 兄弟: PRESCHOOL_GUEST(未就学) / ALUMNI_GUEST(中学生以上の元学童)
- 保護者: PRIMARY_GUARDIAN / SECONDARY_GUARDIAN / EMERGENCY_CONTACT
- 配慮事項・アレルギー・写真公開可否などの属性は全員に持てる
- **個人情報保存期間: 卒所後5年保持 → 自動アーカイブ。削除はテナント管理者操作**(初期値、テナントごとに設定可)

### 機能スコープ(第1版)
- 請求書発行(MONTH / EVENT / VACATION / OUTING、台帳方式、訂正行で整合)
- 請求書は **メール送信 + 保護者ログイン画面で閲覧**(印刷PDFは出力しない)
- **決済機能なし**。入金確認は通帳CSV取込 or 手動入力で消込フラグ更新
- 通知は **メールのみ**(SES)、LINE/Pushは将来検討
- 出席管理(児童・指導員)
- シフト管理(指導員)
- 議事録管理(例会・役員会・運営委員会、保護者にも閲覧可)
- お知らせ配信(学級閉鎖・気象警報・行事案内のメール一斉送信)
- 入所申込・退所手続のWeb化
- レポート(月次売上、未収一覧、児童在籍数推移)
- 規程文書のバージョン管理(就業規則・賃金規程・運営規程など)
- 総会議決のWeb化(委任状・議決権行使書を電子化)
- **指導員給与計算・労務管理**(賃金規程・就業規則準拠、各種手当・賞与・退職金共済・有給管理)
- 行事実行委員機能(参加者名簿/連絡先/アレルギー閲覧、実費精算・按分計算、会計役員承認フロー)

### AI支援機能(MVPでは最優先2項目、段階的拡張)
学童特有の「1年で役員交代」「引継ぎコスト」「指導員連携不足」に直撃する領域を優先:

**◎ 強相性(MVPでも検討)**
1. **引継ぎ文書の自動生成** — 過去の同行事(ファミリーキャンプ等)の実行委員記録・収支・反省点をRAGして、今年の担当者向け引継ぎ書を生成
2. **議事録の要約・議題タグ・決定事項抽出** — 例会・役員会・運営委員会の録音/テキストから要約
3. **保育日誌のサマリと気になる児童の検知** — 日誌横断で週次サマリ生成

**○ 第2フェーズ**
4. 保護者問い合わせ自動応答(規程RAG)
5. シフト自動作成提案
6. 規程改定ドラフト支援
7. アレルギー・配慮事項の見落とし防止
8. 年次総括・保育総括のドラフト

**△ 優先度低**
- 滞納督促文面生成、写真整理、入所説明会Q&A

実装方針: AI機能は独立Lambda + Bedrock(Claude)で疎結合。S3+ベクトルDBのRAG基盤を1つ立てて文書系AIは全部その上に乗せる。

---

## 未解決の論点

- 保存期間「卒所後5年」の初期値は妥当か、施設の運用ポリシーで調整余地あり
- 個人情報保護法の改正動向(2026年時点)の最新確認
- 議事録に **保護者の発言** が記録される場合のプライバシー範囲
- 「行事実行委員」が他世帯の連絡先を閲覧する権限と、プライバシーの境界(本対話では「実行委員は名簿アクセス可」で合意したが、ひまわり実態でどの程度まで開示しているかは要追加ヒアリング)
- 通帳CSVのフォーマット(播州信用金庫東灘支店)を実物確認する必要あり
- SaaS運営者の課金モデル(月額×施設、児童数従量、など)
- AI機能の利用に伴うコスト試算とテナント課金への反映方法

## 対話の生ログ

### Q1. ひまわり自社利用 or 他施設販売型?
**A**: 将来他の学童でも販売提供するシステムのつもりです。

### Q2. 立ち上げのフェーズ感は?
**A**: 半年後に他施設のイメージです。

### Q3. 想定顧客のイメージは?
**A**: 民間学童です児童数100名でも対応できるように。

### Q4. ログインユーザーは誰?(a/b/c)
**A**: B (施設管理者・職員 + 保護者ログインあり)。

### Q5. 保護者ログイン後の機能は?(①〜⑦から選択)
**A**: 全てです。また役割によってはイベントの開催や告知、集計、集金などの処理も必要です。

### Q6. ロール構成は?(SaaS運営者/施設管理者/職員/イベント担当/会計担当/保護者)
**A**: 学童で、１年に一度保護者が役員になったり、ファミリーキャンプのイベント主催者になったりして、各家族や児童に対して集金をすることがあります。また、夏休みに昼食代や、子供だけで映画を見に行くなどのイベント参加時に別途集計して集金することもあります。そのあたりは保護者が主体になって行いますので、役割が付いた保護者や、役員になった保護者には権限が必要なんりますが、１年後にはその役割を外すことがあります。また、役割によっては閲覧が必要な画面があったりするので、保護者に時限的にロールを割り当てる機能が必要です。

### Q7-Q8. 保護者ロールの具体例/プライバシー範囲
**A**: ひまわり総会議案書PDFを提供 — 役員構成・行事実行委員・会議出席分担・保育料表・賃金規程・就業規則・運営規程・基本規約などの実態が明らかに。

### Q9. 指導員給与計算をスコープに?
**A**: (Q11と合わせて回答) その辺りも管理できることが望ましいですね。過去歴も必要だと思うので。議題に応じてすぐに誰もが確認でき、役割があるものが編集可能で、誰が編集したかの礫もわかるようにしなければいけないです。

### AI活用の相性
**A**: AIによる手助けができるよにもしたいです。どこを手助けするかはこれから検討が必要ですが、相性が良さそうなところを教えてもらえば検討します。

### 家族情報の登録範囲
**A**: 児童の家族の情報も登録する必要があります。今後、学童に入ってくる児童がいるか、またイベントに参加する未就学児童や、中学生以降の兄弟がいるかなど、また卒所したが、過去にいた自動の歴なども必要ですね。

### Q10/Q12/Q13. 決済・請求書・通知
**A**: 決済は管理しません。請求書はメールやユーザー画面で確認できるとありがたいです。請求したが入金されていないなどの確認は通帳の突き合わせで確認できれば良いです。通知はメールです。

### Q14a. 機能スコープ
**A**: Q14に関しては全てですね(全機能を実装スコープに含める)。

### Q14b. 個人情報保存期間
**A**: 推奨値「卒所後5年保持→自動アーカイブ、削除はテナント管理者操作」で進める。
