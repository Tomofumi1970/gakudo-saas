# 学童保育向け 請求・会員管理 SaaS — 計画書

> ステータス: **計画段階** (実装着手前)
> 最終更新: 2026-05-11

このドキュメントは、ひまわりクラブHPプロジェクト(本リポジトリ)とは**別プロジェクト**として将来構築する SaaS の仕様メモです。
プロンプトは確定版を本書末尾に保管しています。仕様を詰めるための作業メモを上部に追記していきます。

---

## プロジェクト分離の方針(推奨)

このSaaSは **新規リポジトリで構築すべき** です。

| 比較項目 | ひまわりクラブHP(本リポジトリ) | 学童SaaS(新規) |
|---|---|---|
| 性質 | 単一施設の広報HP | マルチテナント業務SaaS |
| データ | S3上のJSONファイル | DynamoDB(中心) |
| 認証 | 職員ログイン(Cognito既存Pool) | 施設管理者・職員・保護者の多階層 |
| 利用者 | 訪問者(=見込保護者) + 職員数名 | 複数施設 × 多数の保護者 |
| デプロイ | S3静的 + 小さなLambda | CDK で多リソース展開 |
| 料金感 | 月額~500円 | 月額数千円〜(施設数依存) |
| 開発リズム | コンテンツ更新中心 | 機能追加・改修頻繁 |

**結論**: リポジトリ・AWSアカウント・課金とも分離。
リポジトリ名候補: `gakudo-billing` / `gakudo-saas` / `gakudo-platform` など。

## ひまわりクラブHPとの関係

- HPはSaaSの「初期利用施設(ORG_himawari)」のショーケースとして機能可能
- HPの保護者問い合わせフォームから、SaaS導入相談を受けることも将来可能
- ただし**現時点ではゼロ連携**で構わない(疎結合)

## 仕様検討中のメモ

### 開放質問(2026-05-11 対話で確定)

詳細は [docs/spec.md](spec.md) および [docs/talks/2026-05-11_学童SaaS仕様.md](talks/2026-05-11_学童SaaS仕様.md) を参照。

- [x] **SaaSはひまわりクラブが自社利用するもの? それとも他施設にも販売するもの?**
  → 他施設にも販売提供する **マルチテナントSaaS**。フェーズ1: ひまわりで実運用、フェーズ2(約6ヶ月後): 他施設提供開始。想定: 民間学童、1施設児童100名規模まで。
- [x] **認証は施設ごとに分離するか?**
  → **Cognito 共通プール + テナント属性(org_id)** で切り分け。データは `org_id` で完全分離。
- [x] **保護者ログインが必要か?**
  → **必要**(第1版から全機能対象)。請求書閲覧・申込・連絡・イベント参加・実行委員機能まで。
- [x] **決済機能の有無**
  → **管理しない**。既存の口座振替を維持。SaaSは請求書発行までで、入金確認は通帳CSV取込 or 手動入力で消込。
- [x] **紙の請求書印刷 or メール配信**
  → **メール送信 + 保護者ログイン画面でPDF閲覧**。印刷PDFは出力しない。
- [x] **LINE WORKS / LINE通知との連携必要性**
  → **メールのみ**(SES1本)。LINE/Pushは将来検討。
- [x] **出席管理機能の有無**
  → **実装する**(児童・指導員)。
- [x] **レポート機能(月次売上、未収一覧)**
  → **実装する**。月次売上、未収一覧、児童在籍数推移。

### 追加で確定した要件(対話で発見)

- **時限的ロール割当**: 保護者役員・行事実行委員など、有効期間付きのロール
- **過去歴の保持**: 卒所した児童、退任した役員、過去の行事実行委員のすべての履歴を保持
- **監査ログ**: 全エンティティに「誰がいつ何を編集したか」を記録
- **閲覧は広く、編集はロール限定**
- **世帯メンバーモデル拡張**: 児童(PROSPECTIVE/ACTIVE/GRADUATED/WITHDRAWN), 兄弟(PRESCHOOL_GUEST/ALUMNI_GUEST), 保護者(PRIMARY/SECONDARY/EMERGENCY)
- **保存期間**: 卒所後5年保持 → 自動アーカイブ、削除はテナント管理者操作のみ(初期値、テナントごとに設定可)
- **指導員給与計算・労務管理をスコープに含める**(賃金規程・就業規則準拠)
- **議事録・規程文書のバージョン管理**(就業規則・賃金規程・運営規程など)
- **総会議決Web化**(委任状・議決権行使書の電子化)
- **行事実行委員機能**: 参加者名簿閲覧、実費按分計算、会計役員承認フロー
- **AI支援機能**: MVPは「引継ぎ文書生成」「議事録要約」「保育日誌サマリ」を優先

### 確定済み(プロンプト記載分)

- 請求先は世帯(household)単位
- メンバー: 保護者複数・子供複数・ステータス管理
- 料金: 月額 / 外出時(品目別) / 休暇昼食 / イベント(年齢区分単価)
- 請求単位: MONTH / EVENT / VACATION / OUTING
- 台帳(ledger)方式、訂正行で整合確保
- AWS サーバーレス + DynamoDB
- CDK (TypeScript) + Lambda (Python)

## 実装着手用プロンプト(確定版)

```
あなたはAWSサーバーレス/DB設計/実装のプロフェッショナルです。
学童保育向けの請求・会員管理SaaSを、AWSサーバーレス + DynamoDB で設計し、実装の叩き台まで作成してください。
開発環境は VSCode + GitHub、AIは Claude を利用します。TypeScript CDK を前提にしてください(LambdaはPythonでもTSでも可。指定がなければLambdaはPython)。

# 1. ビジネス要件(確定)
- 請求先は「世帯(household)」単位。
- メンバー管理:
  - 世帯(住所/TEL)に保護者(複数)と子供(複数)が紐づく。
  - 子供は小学生の間在籍するが、途中退所もある(status: ACTIVE/WITHDRAWN/GRADUATED)。
  - 未就学児や卒業後中学生など、学童外の参加者もイベントで参加し得る(guest)。
  - 特徴(アレルギー・配慮事項・メモ)を持てる。
- 料金:
  - 月額固定(学童費)
  - 外出時費用:参加した児童のみ。飲み物などの品目(ジュース150円、お茶110円等)を追加でき、当日参加者に課金。
  - 休暇(夏/冬/春)昼食費:その日に来所した児童のみ課金。休暇単位で集計して請求。
  - イベント(キャンプ/スキー等):その都度費用が異なる。年齢区分で単価が変わる例:
    - 大人 5000円
    - 学童在籍の子 3000円
    - 未就学 1500円
    - 卒業後中学生は大人と同額
  - イベントは実費精算があり、余剰金は各参加者の負担割合に応じて返金する。追加徴収が必要な場合も同様に按分する。
- 請求単位:
  - 締めの基本は「月」だが、月額以外(イベント、休暇、外出)は「その単位で請求書を発行」する。
  - つまり請求書は、MONTHだけでなく EVENT / VACATION / OUTING 単位でも発行できる必要がある。

# 2. システム要件
- AWSサーバーレス構成(Cognito, API Gateway, Lambda, DynamoDB, S3, EventBridge, SES など)
- DynamoDBを中心にスケールする設計
- 監査/修正に強い設計:
  - 台帳(ledger)方式:課金/返金は「原票ログ」として1行ずつ永続化し、請求書は集計スナップショットとして生成する。
  - 単価変更が後からあっても過去ログが壊れないよう、台帳に unitPrice と amount を記録する。
  - 原則、削除修正ではなく「訂正行(差額)」追加で整合を取れるようにする。

# 3. あなたが出力する成果物(必須)
A) アーキテクチャ図(テキストでOK)
B) DynamoDBのテーブル設計(キー設計、属性、GSI含む)
   - 施設(ORG)マルチテナント前提
   - 世帯/メンバー
   - 品目カタログ(飲み物、昼食、イベント料金など)
   - 台帳(課金/返金ログ)
   - 請求書(スナップショット)
   - 「請求単位」が MONTH/EVENT/VACATION/OUTING で共通に扱えるキー設計にすること
C) CDK(TypeScript)で DynamoDB テーブルを作るコード(最低限のスタック)
D) 主要API設計(OpenAPIの雛形で良い)
   - メンバー登録/更新/退所
   - 品目追加/更新
   - 課金登録(単発・一括)
   - イベント作成/参加登録/実費確定(返金/追加徴収の按分)
   - 請求書生成(MONTH/EVENT/VACATION/OUTING)
E) 返金按分ロジックの仕様と、ユニットテスト例(PythonでOK)
   - 按分:参加者の見込み負担額(plannedCharge)に比例
   - 端数:合計が一致するよう最大負担者から±1円で調整(明確なルールにする)

# 4. 設計のガイド(守ること)
- DynamoDBはホットパーティションを避ける(施設×請求単位×期間で分散)
- 取得パターンを明確にし、それに合わせてGSIを設計する
  - 世帯別の請求明細一覧
  - 子供/参加者別の履歴
  - 在籍児童一覧(ACTIVE)
- 後から施設が増える前提でORGで完全に分離できること
- 可能ならコスト最小(オンデマンド or 適切なキャパシティ)

# 5. 初期データ例(必須)
- 飲み物:ジュース150、お茶110
- 昼食:例として300(可変にできる設計)
- イベント料金:adult=5000, afterschool=3000, preschool=1500, juniorHigh=adult

# 6. 出力形式
- 章立てで読みやすく
- まず仕様前提→次にテーブル→次にAPI→次にCDKコード→次にテスト
- コードはコピペで動く程度に具体的に
```
